## 1. Project Goal

Build a mobile companion + server-side terminal that share Claude Code sessions seamlessly, so the user can drive the same agent from their phone during commute and from a desktop terminal at home/office without manual session juggling.

**Primary use case:** Daily commute (~2 hours/day). On the way out the door, user drives ongoing CC tasks from phone; once at desktop, switches to terminal and continues without losing state.

**Final target audience:** ordinary users (not just the author).

- **v0.1** ships as self-host + open source, target = technically literate self-hosters
- **v0.2+** lands the multi-user/multi-machine commercial path (see §13)
- All v0.1 design choices are evaluated for two criteria: (a) v0.1 works correctly; (b) v0.2+ doesn't require a ground-up rewrite

---

## 2. Goals & Non-Goals

### Goals (v0.1)

1. Same Claude Code session accessible from phone (PWA) and desktop SSH terminal, real-time bidirectional sync
2. **Server-owned CC processes** — sessions are not bound to whichever device created them; either device can attach to any session. **Note:** in v0.1, daemon-crash CC survival is **unproven** (see §3 Bet 1, §11 PoC #3). Default durability contract: daemon crash *may* terminate the live CC; session continuity is recovered via JSONL + `tether resume`, not guaranteed in-process survival.
3. Mobile UX adequate for commute use — message stream, code viewer, send messages, approve/deny tool calls
4. Terminal client gives full native CC TUI via custom raw-mode client connecting to daemon
5. **Authenticated transport encryption** with explicit, accurate threat model (NOT "E2E"; see §9)
6. Self-hostable — single npm package + node-pty native dep; no systemd/root required
7. Web Push notifications (best-effort; iOS limitations explicitly documented)
8. **Enforceable single active-writer guarantee** — daemon owns PTY input gate; lock is mechanism, not policy (see §11)
9. **Deterministic state model** — JSONL is single source of truth; PTY and PWA chat view are derived renderings; replay uses CC message UUIDs (see §6)
10. Forward-compatible schema — v0.2+ multi-user/multi-machine path concretely specified (see §13)

### Non-Goals (v0.1)

- ❌ Multi-user accounts — single user assumed in implementation, but reserved at protocol/storage layer
- ❌ Multi-machine federation — one server only in v0.1, machineId reserved
- ❌ Native iOS/Android app — PWA only
- ❌ Artifact storage (persistent markdown documents)
- ❌ Voice input / output
- ❌ Shell RPC (`session.bash` for arbitrary commands from phone) — too much scope, defer
- ❌ Full project file tree on mobile — only changed-files sidebar
- ❌ Hosted relay infrastructure
- ❌ NAT traversal — user provides their own connectivity (Tailscale, Cloudflare Tunnel, public IP, etc.)
- ❌ systemd installer — daemon runs in userspace, no root required
- ❌ External `dtach` use — daemon owns PTYs via node-pty; users cannot bypass via raw `dtach -a`

## 3. Risks & Open Architectural Bets

This section is intentionally placed before the architecture so that any reader understands the load-bearing assumptions before evaluating details.

### Bet 1 (highest risk): the custom PTY ownership / survival / attach stack around node-pty

This is the **single biggest residual risk in v0.1** (post review #3). Replaces "dtach + JSONL coherence" from v3.

**Claim:** Daemon owning the PTY master fd via node-pty plus a ~600-LOC custom raw-mode TTY attach client delivers correct multi-client semantics: enforceable single-writer, lossless output fan-out, crash-consistent replay, AND graceful behavior on daemon-restart.

**Why this is the biggest risk:**
- It is the most novel code in the design (combines lifecycle, raw terminal handling, crash recovery, backpressure)
- Failure here directly breaks the core promise of seamless mobile↔terminal handoff
- Multiple new components must work together: node-pty wrapper, ring buffer + fan-out, input gate, raw-mode TTY client, daemon-survival mitigation

**Specific sub-risks:**
- node-pty native module breakage on Alpine/musl/ARM container variants
- Custom raw-mode terminal client mishandling escape sequences, **bracketed paste / paste bursts**, **alternate screen buffer / TUI transitions**, **raw-mode restore on abnormal exit** (the three most likely-to-bite-early failure modes)
- **Daemon-death CC orphaning:** with daemon as parent of CC, daemon SIGKILL may HUP the CC slave PTY; `setsid` alone may not suffice (process-group help, but slave-side hangup can still fire). Supervisor wrapper with PTY/socket handoff is the alternative but is implementation-heavy. **PoC #3 must establish which mitigation works; if neither, default contract is "daemon crash may kill CC, recover via `tether resume`".**
- JSONL writes may be partial during CC crashes; CC may write duplicate UUIDs on hook retries
- Output fan-out under slow client → fast CC produces backpressure / memory growth (mitigated by drain-eager policy in §11)

**Validation gate:** **Week 1 PoCs (§11) must prove all four scenarios** before we commit. Any failure → architecture revision before continuing.

### Bet 2: CC's hook + JSONL contract is stable enough

**Claim:** We can rely on CC's `SessionStart` hook firing reliably, message UUIDs being unique-per-event, and JSONL format being stable across our supported version range.

**Mitigation:**
- **Pinned CC version range** declared in `package.json`; daemon hard-fails on startup if local CC is outside range
- Vendored CC integration code under our explicit ownership (§10)
- CI runs against the pinned range; out-of-range is an explicit user choice with no support promise
- We do NOT promise specific SLA for new CC version compatibility — users wait for tested release

### Bet 3: WebTransport is mature enough in 2026 for production

**Claim:** Browsers and operational tooling (proxies, CDN, certs) handle WebTransport well enough that we can default to it.

**Reality check (April 2026):** Safari 26.4+ supports WT (Baseline status). Chrome/Firefox/Edge support since 2022. **But many self-host environments don't pass through QUIC reliably.**

**Mitigation:** WebSocket over TLS is **co-equal**, not "fallback". Client tries WT first, falls back to WS within 3 seconds. We expect a substantial portion of self-hosters end on WS; we don't penalize them.

### Bet 4: iOS Safari Web Push is reliable enough

**Claim:** Push notifications via Web Push API on iOS deliver promptly enough for the "agent finished, look at phone" UX.

**Reality:** iOS pushes are sometimes delayed (minutes), occasionally silently dropped, and require the PWA to be installed via "Add to Home Screen". Android Chrome is more reliable.

**Mitigation:** PWA's onboarding screen explicitly states the limitations; UI never claims "real-time" notifications. If iOS unreliability proves to be a deal-breaker during dogfood, v0.2 priority shifts to native iOS app.

### Bet 5: Forward-compat without rewrite to v0.2

**Claim:** Reserving `userId`/`machineId` fields, using user-scoped storage paths, and per-session data keys is enough to make v0.2 multi-user a non-breaking change.

**Honest assessment:** Reserved fields **reduce** migration pain but **do not eliminate it**. v0.2 will require: (a) auth flow extension (login on top of pairing); (b) ACL layer; (c) storage migration to populate userId; (d) protocol minor version bump. The cost is bounded if we get the schemas right now (§13).

### Component breakdown

#### `tether-cli` daemon (Node.js + node-pty, ~5500 LOC)

- **`src/transport/`** — WT + WS dual endpoints; common-subset abstraction
- **`src/auth/`** — pair, access token, refresh-token family with grace window, revocation, device binding
- **`src/session/`** — manager, mode-switch lock state machine (enforceable), JSONL watcher, replay engine using CC message UUIDs
- **`src/pty/`** — node-pty wrapper, output ring buffer (512 KB per session), fan-out scheduler, input gate
- **`src/cc/`** — vendored CC integration (§10)
- **`src/push/`** — VAPID keypair, Web Push subscription store, payload signing
- **`src/crypto/`** — XChaCha20-Poly1305 (libsodium secretbox); libsodium box for asymmetric
- **`src/http/`** — static asset server for PWA, healthz, metrics
- **`src/cli/`** — command dispatch (see §15 full command list)
- **`src/storage/`** — `~/.tether/users/<userId>/...` (v0.1 always uses `userId="default"`)
- **`src/observability/`** — structured logging, metrics, audit log

#### `tether attach` raw-mode TTY client (~600 LOC, ships with the npm package)

A separate executable that connects to daemon's Unix socket. **v0.1 supports Unix-socket-only**; remote terminal access is achieved by SSH-tunneling (user runs `ssh server -t tether attach <id>`) — the tether-attach binary still talks to a Unix socket, just on the server side after SSH puts you there. Direct WT/WS terminal mode (tether-attach over network) is **deferred to v0.2+**.

Responsibilities:
- Switch local TTY to raw mode; restore on exit
- Connect to daemon, send `attach { sessionId }`
- Bidirectional pipe: local stdin → socket; socket → local stdout
- Forward SIGWINCH (window resize) to daemon → daemon ioctls PTY size
- Capture special key sequences (`Ctrl+\\` family) for client-side commands (detach, force takeover, status)
- Paste-mode bracket sequences passed through unmodified
- Clean exit on EOF or Ctrl+\\

#### `tether-app` PWA (React + Vite, ~3000 LOC after vendoring components)

Routes:
- `/` — session list (across all machines, but v0.1 only one machine)
- `/session/:id` — chat view + active mode-switch lock state
- `/session/:id/file?path=` — file viewer (path-restricted; see §16)
- `/session/:id/files` — git-changed sidebar
- `/pair` — QR pairing flow
- `/settings` — server URL, push permissions, theme, paired devices

Vendored from happy-app/sources/components/:
- `SimpleSyntaxHighlighter` (321 LOC)
- `PierreDiffView` (250 LOC, uses `@pierre/diffs`)
- `CodeView`, `FileIcon`, `FilesSidebar`, `InlineFileDiff`

State: lightweight reactive store (~500 LOC).

## 5. Wire Protocol

### Transport selection — co-equal

```
Client connects:
  1. Try WebTransport (URL: tether-server:7777/wt)
  2. If fail or 3s timeout → WebSocket over TLS (URL: tether-server:7777/ws)
  3. Show banner in PWA on connection establishment: "Connected via WT" or "Connected via WS"
```

v0.1 uses **only the common subset of WT and WS capabilities**. We do NOT use WT's independent stream multiplexing, connection migration, or datagrams. Single ordered binary stream both sides; treat as equivalent.

### Message envelope

```typescript
type Envelope = {
    v: 1;                          // protocol version
    id: string;                    // unique cuid2 per envelope
    type: MessageType;
    userId: string;                // RESERVED FOR v0.2+; v0.1 always "default"
    machineId: string;             // RESERVED FOR v0.2+; v0.1 set to local machine id
    sessionId: string | null;      // null for non-session messages (auth, etc.)
    timestamp: number;             // ms since epoch, sender's clock
    nonce: Uint8Array;             // 24 bytes for XChaCha20-Poly1305
    payload: Uint8Array;           // XChaCha20-Poly1305 ciphertext + tag
}
```

### Message types

```typescript
type MessageType =
    // bidirectional control
    | 'auth-pair-request'
    | 'auth-pair-response'
    | 'auth-token-refresh'
    | 'rpc-call'
    | 'rpc-response'
    | 'rpc-error'

    // server → client (push, ordered, with eventUuid + seq for ack)
    | 'session-message'             // new chat message; carries CC's own message UUID
    | 'session-state'               // mode change, thinking state, lock holder change
    | 'session-list-update'         // session created/removed/renamed
    | 'session-replay-cursor'       // server tells client: caught up to eventUuid X (seq N)
    | 'session-replay-error'        // cursor invalidated (stale); client must full re-sync
    | 'session-pty-output'          // raw PTY bytes for tether-attach clients (terminal only)

    // client → server
    | 'session-spawn'
    | 'session-input'               // bytes to inject into CC stdin (PWA only sends text; tether-attach sends raw)
    | 'session-attach'              // request stream subscription; carries sinceEventUuid for replay
    | 'session-detach'
    | 'session-kill'
    | 'session-lock-request'
    | 'session-lock-release'
    | 'session-resize'              // SIGWINCH event from terminal client → forwarded to PTY ioctl

    // RPC methods (in payload of rpc-call)
    // - session.readFile  → see §16 (scoped to session.cwd by default)
    // - session.gitStatus
    // - session.setMode
    // - session.approveTool
    // - session.lockStatus
    // (NO session.bash in v0.1 — scope cut)
```

### Replay addressing

Client tracks `lastSeenEventUuid` per session. On reconnect:
1. Client sends `session-attach { sessionId, sinceEventUuid }`
2. Server's replay engine looks up `eventUuid` in its UUID→jsonl-offset index
3. Server replays events with monotonic `seq` starting after that point
4. Server sends `session-replay-cursor { eventUuid, seq }` when caught up

`seq` is a per-attach monotonic counter for in-order delivery. `eventUuid` is the stable identity used for resume and dedup.

## 6. State Model & Replay

This addresses the most-fix #1 from review #1 (one source of truth) and #3 from review #2 (replay correctness).

### Authoritative model

```
                           CC process (one per session)
                           │
                  ┌────────┼────────┐
                  ▼        ▼        ▼
              [PTY out]  [stdin]  [JSONL writes]
                  │                    │
                  ▼                    ▼
        ┌────────────────┐    ┌──────────────────────┐
        │ daemon node-pty│    │ daemon JSONL watcher │
        │  master fd     │    │                      │
        └────┬───────────┘    └──────────┬───────────┘
             │ (raw bytes)               │ (structured events with UUIDs)
             ▼                           ▼
       ┌──────────────┐         ┌──────────────┐
       │ ring buffer  │         │ replay engine│
       │ + fan-out    │         │ + UUID index │
       └──────┬───────┘         └──────┬───────┘
              │                        │
              ▼                        ▼
         ┌──────────────┐    ┌───────────────────┐
         │ tether attach│    │ PWA chat view     │
         │ (terminal)   │    │ (deserializes msg)│
         └──────────────┘    └───────────────────┘
```

**Rules:**

1. **JSONL is the single source of truth for structured state.** Chat history, tool calls, agent state, mode all derive from JSONL. The replay engine reads JSONL exclusively.
2. **PTY is rendering for terminal clients only.** Bytes flowing through PTY are NOT used to derive structured state — they exist solely to render CC's TUI to terminals.
3. **PWA only sees JSONL-derived events.** It never receives raw PTY bytes.
4. **Terminal sees PTY only.** It does not parse JSONL itself.

### Event identity (CC's own UUIDs)

CC's JSONL format includes a UUID on **most replay-relevant event records** — user messages, assistant messages, tool uses, tool results, attachments, system turn-duration. **Some metadata records do NOT carry a UUID** (e.g., permission-mode lines, file-history-snapshots, last-prompt summaries). We use the embedded UUIDs as stable event identifiers; UUID-less metadata records are not "events" for replay purposes — they are state hints derived alongside, and the replay engine handles them as session-state updates rather than `session-message` items.

We refer to these as **event UUIDs** (broader than just message UUIDs).

Replay engine maintains:

```
sessions/<id>/index.json
{
  "events": [
    { "uuid": "01J...", "jsonlOffset": 0,    "byteLen": 412, "kind": "user-message" },
    { "uuid": "01K...", "jsonlOffset": 412,  "byteLen": 856, "kind": "assistant-message" },
    ...
  ]
}
```

This index is **rebuilt on every daemon start by reading the JSONL from offset 0**. We do not rely on the index being durable — we rely on JSONL being durable (CC writes it). For sessions with very large histories (>50k events), full rebuild may take seconds; future optimization (out of v0.1 scope) is checkpointed streaming rebuild persisting last-known-good byte offset + index snapshot.

### Duplicate detection

Dedup key is **uuid alone** (not a tuple with offset — duplicates by definition occur at different offsets).

If CC writes a duplicate event (e.g., hook retry produces same UUID twice):
1. Replay engine maintains an in-memory `Set<uuid>` of known events
2. While parsing, if line's uuid is already in the set: mark the line as `{ ..., "duplicate": true }` in `index.json`, do NOT broadcast a `session-message`, increment metric
3. Metric: `tether_replay_duplicate_detected_total` (no sessionId label — see §17)

### Torn write handling

If the last line of JSONL is invalid JSON (truncated, mid-write):
1. Replay engine stops parsing at that line; advances watermark only up to last known-good event
2. **Server never advances replay watermark past unverified complete line** — this addresses review #2's #3-c
3. On next CC restart (`tether resume <id>`), CC truncates the bad line and continues; new valid lines extend the JSONL
4. Client connecting with old `lastSeenEventUuid`: if the UUID still exists in the index, replay from there; if not (rare — CC overwrote in a way that lost the line), server returns `session-replay-error { reason: 'cursor-invalidated' }` (see §5) and client must do full re-sync
5. Metric: `tether_jsonl_torn_writes_total` (no sessionId label)

### What about input from PWA?

When PWA sends `session-input { text: "continue" }`:
1. Daemon verifies PWA holds the write lock (§8). If not → reject with `rpc-error`.
2. If lock held: write `text + "\n"` to PTY's master fd via node-pty's `write()` method
3. CC receives bytes on stdin, processes, writes JSONL entry with new event UUID
4. JSONL watcher sees new line, parses, broadcasts as `session-message` to all attached clients
5. PWA waits for the JSONL echo before showing the message as confirmed (NO optimistic render)

**Latency:** ~100–300ms from send to confirmed echo. Acceptable for human-paced chat.

### What about input from tether-attach (terminal)?

1. Local TTY in raw mode captures user keystrokes
2. tether-attach sends `session-input { bytes: <raw> }` to daemon
3. Daemon checks lock (terminal's implicit lock from typing — see §8). If held by terminal → write to PTY master fd. If held by mobile → drop bytes, send back a control message to display banner in client TTY.
4. PTY echoes input → output ring buffer → fan-out → tether-attach renders to local TTY
5. CC eventually writes JSONL → broadcast to PWA (and any other attached clients)

**Important:** tether-attach does NOT directly write to PTY. All bytes flow through daemon's input gate. This is what makes the lock enforceable.

## 7. Authentication & Pairing (with token family + grace window)

This addresses review #2's must-fix #4.

### Token model

| Type | Lifetime | Bound to | Notes |
|---|---|---|---|
| **Pairing token** | 60s | URL nonce | Single-use, deleted after consumption |
| **Access token** | 1h | Device pubkey hash | Sent with every transport message |
| **Refresh token** | 30d (rolling) | Device pubkey + token-family-id | See rotation logic below |

### Pairing flow (unchanged from v3)

1. `tether daemon` first run: generate VAPID keypair, libsodium box keypair, store in `~/.tether/users/default/keys.json` (mode 600)
2. `tether pair` issues 60s single-use pairing token + QR code with server pubkey + token
3. PWA scans, generates own keypair, POSTs `/v1/pair` with `{ devicePubkey, machineFingerprint, pairToken }`
4. Daemon validates pairToken (single-use), exchanges via X25519, derives shared secret
5. Daemon issues `{ accessToken, refreshToken, tokenFamily, serverPubkey }`; persists `{ deviceId, devicePubkey, currentRefreshTokenHash, tokenFamilyId }` to `devices.json`
6. PWA persists in IndexedDB (origin-isolated)

### Refresh-token rotation with family + grace window

This is the heart of the must-fix.

**Critical implementation requirement: refresh rotation MUST be serialized per token-family** — either via a per-family mutex in daemon memory OR via atomic compare-and-swap at the storage layer. Without this, two concurrent refreshes can both observe `R0.status == current` and both mint distinct successors, breaking idempotency. The state machine below assumes this serialization is in place.

**Grace-window timing is server-clock authoritative.** Client-clock skew does NOT extend grace. Documented client guidance: on uncertain network state, retry refresh **promptly**; do not wait beyond expected RTT. If grace is missed, re-pair is required.

```
Token family:
  familyId: cuid2
  generation: monotonic counter (starts at 0 on pair)
  members: list of (refreshTokenHash, generation, issuedAt, status, successorTokenHash?)
    where status ∈ {'current', 'grace', 'revoked'}

Refresh request from device with refreshToken R, devicePubkey K:
  1. Acquire per-family lock for family containing hash(R)
     - Not found in any family → reject
  2. If family F is revoked → reject; require re-pair
  3. If R.status == 'current':
       - Mint R+1 at generation G+1 with status='current'
       - Set R.successorTokenHash = hash(R+1)
       - Mark R as status='grace', graceExpiry = now + 30s (server clock)
       - Issue new accessToken bound to K
       - Return { accessToken, refreshToken: R+1 }
  4. If R.status == 'grace':
     a. If now > graceExpiry: revoke entire family; require re-pair
     b. If R was rotated by SAME devicePubkey K: return idempotent successor
          - Look up R+1 via R.successorTokenHash, return same token
          - Handles: lost-response retry, two-tab race on same device
     c. If R was rotated by DIFFERENT devicePubkey: divergence detected
          - Revoke entire family; require re-pair on legitimate device
  5. If R.status == 'revoked':
       - Reject; require re-pair
  6. Release per-family lock
```

**Bricking risk:** A single packet loss with delayed retry (>30s after rotation) results in family revocation and forced re-pair. We accept this for v0.1; clients are expected to retry within seconds of network recovery.

**Stolen-token damage window:** If an attacker's refresh hits the server *before* legitimate device's, attacker gets a fresh access token with **1-hour lifetime**. Family revocation prevents *future* refreshes but does NOT retract the already-issued access token unless **access tokens are checked against family-revocation state on each request**. v0.1 adopts this stateful access-token check (cheap: hash lookup per envelope) — limits damage to ≤ N seconds (the time between revocation and the attacker's next request).

Properties:
- **Lost response retry within 30s:** legitimate device retry returns the same successor; no bricking
- **Two-tab race within 30s:** both calls return same successor; both tabs work
- **Stolen token used after legitimate device:** legitimate device's earlier refresh promoted R to grace, attacker presents R, sees grace, gets the same successor that legit device already got. Attacker sees what legit device sees (still attacker has compromised tokens). HOWEVER: when attacker's R+1 is then used out of band, family revokes on detected divergence — see below
- **Stolen token used before legitimate device:** attacker rotates R to R+1; legitimate device uses old R, sees grace, but `idempotent successor` requires recognizing the same caller. We use `devicePubkey` as caller identity: same pubkey → same idempotent successor; different pubkey on grace token → revoke family

### Divergence detection

Same-device-pubkey-on-grace returns idempotent successor (path 4b above). Different-pubkey-on-grace revokes the family (path 4c).

Idempotency key for grace-window successor lookup: `(refreshTokenHash, devicePubkey)`. Different pubkey hits the divergence path.

### Revocation

- `tether devices` lists paired devices
- `tether revoke <deviceId>` sets family status='revoked'; subsequent token use rejected
- `tether revoke --all` rotates server keys (generates new VAPID, new box keypair); all devices must re-pair

### Clock drift tolerance

- Token expiry checks allow ±5min skew between server and client
- Pairing token expiry uses server clock only (60s window narrows skew exposure)
- Documented requirement: daemon host must have NTP-synced clock; we log a warning if observed skew > 60s

### Why not OAuth in v0.1

v0.1 single-user; pairing IS identity. v0.2 adds login on top — pairing remains the device-authentication layer.

## 8. Mode-Switch (Single Active Writer, Now Enforceable)

The lock is now a **mechanism**, not a policy, because daemon owns PTY input gate.

### State

```
Per session:
  holder: 'terminal-{deviceId}' | 'mobile-{deviceId}' | null
  lastWriteAt: timestamp ms
  takeoverHistory: [{ ts, fromHolder, toHolder, kind }]   // audit log
```

Auto-release: `lastWriteAt + 60s < now` → `holder := null`

### Acquisition rules

- **Terminal acquisition (implicit):** tether-attach client sends `session-input` with bytes. Daemon's input gate evaluates:
  - If `holder == null` OR `holder == this terminal-deviceId`: accept; update lastWriteAt; if was null, set holder
  - If `holder == 'mobile-X'` AND lastWriteAt > 60s ago: take over; set holder; emit `session-state { holder }` to all clients
  - If `holder == 'mobile-X'` AND lastWriteAt within 60s: **reject bytes**, do NOT write to PTY; respond with control message that instructs tether-attach to display banner "Mobile holds lock. Press Ctrl+\\ Ctrl+T to force takeover."
- **Mobile acquisition (explicit):** PWA sends `session-lock-request`. Daemon evaluates:
  - If `holder == null` OR `holder == this mobile-deviceId`: granted
  - If `holder == 'terminal-X'` AND lastWriteAt > 60s ago: granted
  - Otherwise: rejected, response carries current holder for UI display
- **Force takeover (either side):** explicit message `session-lock-force-takeover`; logged to audit log; current holder informed via `session-state` event. **Both sides see the takeover happen.**

### Why terminal is implicit but mobile is explicit

- Terminal: keystrokes are unambiguous intent. Forcing user to "click to acquire" before typing is unnecessary friction.
- Mobile: opening PWA does not imply intent to type. A user just checking progress should not silently steal lock from terminal.

### Mobile UI states

| Lock holder | PWA input area |
|---|---|
| `null` | Active, placeholder "Type message…" |
| `mobile-self` | Active, holding-lock indicator |
| `mobile-other` (v0.2+ multi-device per user) | Disabled, "Other device active. [Take over]" |
| `terminal` | Disabled, "Terminal is active. [Take over]" |

### Audit log

Every lock state transition logged:

```
~/.tether/users/default/sessions/<id>/lock.log
2026-04-26T10:00:00Z grant terminal:DEV1 (implicit, was null)
2026-04-26T10:30:00Z release terminal:DEV1 (idle 60s)
2026-04-26T10:30:15Z grant mobile:DEV2 (explicit)
2026-04-26T11:00:00Z force-grant terminal:DEV1 from mobile:DEV2
```

Exposed via `tether locks <sessionId>`.

### What about force takeover during input race?

```
Walkthrough:
  T0: terminal holds lock, typing slowly
  T1: mobile clicks "Force takeover"
  T2: daemon receives session-lock-force-takeover from mobile
  T3: daemon sets holder := mobile; emits session-state to terminal client
  T4: terminal client renders banner "Mobile took over. Press Ctrl+\\ Ctrl+T to reclaim."
  T5: terminal user types more bytes
  T6: daemon's input gate: holder is mobile, terminal bytes rejected, control banner re-displayed
```

Single coherent state at every moment. No drift between "what daemon thinks" and "what CC receives".

## 9. Encryption (XChaCha20-Poly1305)

This addresses review #2's must-fix #5.

### Why XChaCha20-Poly1305 (not AES-GCM)

- 192-bit nonce (24 bytes) — random nonce collision probability is cryptographically negligible (2^-160 at 2^32 messages)
- libsodium native (`crypto_secretbox_xchacha20poly1305`)
- AEAD: encryption + authentication in one primitive
- Constant-time on all platforms (no AES-NI dependency)
- Same security as AES-GCM but no nonce-uniqueness footgun

### Crypto primitives

- **Asymmetric key exchange:** libsodium X25519 (`crypto_box_*`)
- **Symmetric AEAD:** XChaCha20-Poly1305 IETF construction. **Correct libsodium API:** `crypto_aead_xchacha20poly1305_ietf_encrypt` / `crypto_aead_xchacha20poly1305_ietf_decrypt`. (Note: `crypto_secretbox_*` is XSalsa20-Poly1305, NOT XChaCha20 — do not use.)
- **Nonce:** 24 bytes random per message; uniqueness scope = per-key lifetime
- **Key derivation:** HKDF-SHA-256 (or libsodium's `crypto_kdf_*` family) from shared secret + per-session salt
- **MAC:** Poly1305 tag built into AEAD

### Key hierarchy

```
Server master secret (random 32 bytes, ~/.tether/users/default/master.key)
  ↓ HKDF
(used to derive subsidiary keys for at-rest encryption — v0.2+; v0.1 unused at rest)

Per-session data key (random 32 bytes generated at session creation, NOT derived from master)
  ↓ XChaCha20-Poly1305 IETF
Encrypted session payloads on wire (24-byte random nonce)

Server libsodium box keypair
  ↓ X25519 with PWA's pubkey at pair time
Shared secret (per device)
  ↓ HKDF
Authentication channel key (encrypts auth/control envelopes)
```

### Per-session data key wrapping

When a session is created, daemon generates a random 32-byte session key. This key is encrypted with each paired device's shared secret (libsodium box) and stored in `sessions/<id>/keys.json`. PWAs unwrap their own copy on attach. Session keys are random per session; they are NOT deterministically derived from the master, so daemon restart does NOT reset key lifetime — the persisted random key continues in use.

### Key versioning (for replay decryption after rotation)

When a session key rotates (on `tether resume`), the new key is appended to a versioned store:

```
sessions/<id>/keys.json
{
  "current": { "version": 3, "key_wrapped_per_device": {...}, "createdAt": ... },
  "history": [
    { "version": 2, "key_wrapped_per_device": {...}, "createdAt": ..., "retiredAt": ... },
    { "version": 1, "key_wrapped_per_device": {...}, "createdAt": ..., "retiredAt": ... }
  ]
}
```

Encrypted envelopes carry an explicit `keyVersion` field (added to `Envelope`); receiver looks up the matching key from history. v0.1 retains all key versions for the session's lifetime (key material is small; no eviction).

### Key rotation

- **Per-session data keys:** rotated on session restart (`tether resume`); old key retained in `keys.json` history for replay decryption
- **Server master secret:** rotated only on `tether revoke --all`
- **No automatic rotation** in v0.1 — accepted simplification; v0.2+ may add scheduled rotation

### Nonce uniqueness scope

Per-key. With 192-bit random nonce and per-session keys (typical key lifetime ~days to months), collision is statistically impossible.

### Replay protection

Each envelope includes `(seq, sessionId, timestamp)`:
- `seq <= last_seen_seq` → reject
- `|now - timestamp| > 5min` → reject

### What about JSONL files at rest?

**v0.1: not encrypted at rest.** They live at CC's own location (`~/.claude/projects/<hash>/<uuid>.jsonl`, mode 600). Trust boundary is "your server"; encrypting at rest only defends against host-level snooping, which is out of scope.

**v0.2+:** consider per-session at-rest encryption when multi-user lands.

### Threat model (explicit)

| Adversary | Defense |
|---|---|
| Passive network sniffer | XChaCha20-Poly1305 AEAD ✓ |
| Active MITM with valid CA cert | Pinned server pubkey post-pairing ✓ |
| Stolen access token | 1h lifetime + device binding ✓ |
| Stolen refresh token (concurrent with legit device) | Family + grace + divergence detection ✓ |
| Stolen device (phone) | User revokes via `tether revoke`; family invalidated ✓ |
| Compromised server | Out of scope — adversary has plaintext access by design |
| Malicious browser extension reading IndexedDB | Out of scope (browser security boundary) |
| Replay attack | Seq + timestamp ✓ |
| Downgrade attack (force WS instead of WT) | WS is also TLS+AEAD; no security weakening |
| Nonce collision DoS | 192-bit nonce makes this cryptographically impossible |
| State exhaustion DoS | Rate limit pair attempts; rate limit token refresh ✓ |

## 11. PTY Multiplexing (node-pty + custom client protocol)

### Architecture

```
node-pty.spawn('claude', ['--session-id', uuid, ...]) → PTY master fd in daemon
                                                              │
                                                ┌─────────────┼──────────────┐
                                                │             │              │
                                                ▼             ▼              ▼
                                          ┌──────────┐  ┌──────────┐   ┌──────────┐
                                          │ output   │  │ output   │   │ JSONL    │
                                          │ ring buf │  │ ring buf │   │ watcher  │
                                          │ → client │  │ → client │   │          │
                                          │   A (raw)│  │   B (raw)│   │          │
                                          └──────────┘  └──────────┘   └──────────┘

Client A (tether-attach via Unix socket)        Client B (another tether-attach)
        ↓ raw bytes from local TTY                       ↓
              session-input { bytes }                          session-input { bytes }
        ↓                                                       ↓
  daemon input gate (lock check) ─────────────────────────────────
                          │
                          ▼ (if accepted)
                    PTY.write(bytes) → CC stdin
```

### Output ring buffer + fan-out

**Critical invariant:** PTY master fd reads always drain eagerly into the daemon-side ring buffer. Client fan-out happens **asynchronously** off this buffer. Slow clients NEVER back-pressure PTY reads, which means CC itself is never stalled by a slow client.

Concretely:
- A dedicated PTY-read loop drains the master fd into the ring as fast as the OS delivers bytes
- Per-client send queues are populated from the ring; each client's queue is drained at its own pace
- If a client's send queue exceeds 1 MB, daemon drops output for THAT client only, emits `session-state { clientLagging: true }`, increments `tether_pty_output_drops_total{reason=client_lag}`. Other clients and the PTY-read loop are unaffected.
- Ring buffer per session: 512 KB (sized for "last visible screen plus moderate scrollback"; larger sizes deferred to v0.2 if needed for use cases like long file dumps)
- New attaches receive the ring's contents on connect (renders last screen of TUI immediately)

### Input gate

- All input arrives via `session-input` envelopes (PWA over WT/WS, terminal over Unix socket)
- Gate checks `(senderType, senderDeviceId, lockHolder)` against rules in §8
- Accepted bytes → `pty.write(bytes)`
- Rejected bytes → respond to sender with control message; for terminal client, this becomes a banner overlay

### PoC blockers (Week 1 gates)

These must pass before continuing past Week 1.

#### PoC 1: node-pty multi-fanout under load

**Test:** Spawn CC via node-pty. Connect 3 simulated clients (raw byte sinks). Generate continuous PTY output (e.g., agent doing work). Verify:
- All 3 clients receive identical byte streams
- TUI renders correctly when one client's local TTY replays the bytes
- Backpressure: artificially slow one client; verify others unaffected

#### PoC 2: Lock-enforced input

**Test:** Spawn CC. Connect terminal-A and mobile-B. Sequence:
- terminal-A sends `session-input "echo hi\n"` → verify CC executes
- mobile-B requests lock → granted
- terminal-A sends bytes again → verify NOT delivered to CC; terminal-A receives banner control
- mobile-B sends `session-input "echo bye\n"` → verify CC executes
- 60s elapse with no input
- terminal-A sends bytes → verify accepted (auto-released lock)

#### PoC 3: Daemon restart recovery

**Test:** Spawn CC. Kill daemon (SIGKILL). CC continues (it's a child of daemon... wait — when daemon dies, does CC die too?).

**Open issue to resolve in PoC:** With node-pty, daemon is the parent of CC. If daemon dies, CC becomes orphaned and may receive SIGHUP, depending on PTY semantics. Two mitigations:
- (a) Use `setsid` so CC has its own process group, immune to PTY HUP
- (b) Use `tetherd-supervisor` lightweight wrapper that re-spawns daemon if it dies, preserving CC PTY via Unix socket handoff

PoC #3 evaluates which mitigation is robust and reproducible. If neither fully works, we accept **CC dies on daemon crash; on restart, daemon detects dead session and offers `tether resume <id>` to spawn a new CC against existing JSONL**.

#### PoC 4: Crash-consistent replay (NEW; addresses review #2 must-fix #3 derisk)

**Test:** Spawn CC. PWA attaches and processes 100 messages. Introduce random failures:
- Kill CC mid-write
- Kill daemon at random points
- Drop network mid-stream to PWA
- Restart everything

**Acceptance criteria (extended after review #3):**

a. **No event lost** — PWA's `lastSeenEventUuid` always corresponds to a valid event in the rebuilt index, OR client receives `session-replay-error { reason: 'cursor-invalidated' }` and successfully full-resyncs without missing events thereafter
b. **No semantic event delivered twice** — dedup by UUID alone (not tuple)
c. **No content mismatch** — same UUID → same content across replays
d. **Hook-server vs JSONL-watcher race covered** — if SessionStart hook fires before/after JSONL line is durably written, daemon converges to consistent state. Test: inject delays between hook firing and JSONL flush.
e. **Cross-restart ordering preserved** — events delivered in same order before and after restart; client can reconcile based on UUID identity even if seq numbers reassign
f. **Cursor-invalidated path actually works end-to-end** — explicitly test the case where client's `lastSeenEventUuid` is no longer present in rebuilt index; verify `session-replay-error` is sent and client full-resync succeeds
g. **Metrics fire correctly** — `tether_replay_duplicate_detected_total`, `tether_jsonl_torn_writes_total`, `tether_session_attach_stale_cursor_total` all increment when expected, zero otherwise

If **any PoC fails:** architecture revision. PoC 1 / PoC 2 failure = node-pty isn't the right tool, consider falling back to dtach with advisory lock and explicit downgrade. PoC 4 failure = need stronger event-id strategy (e.g., content-hash plus UUID, or daemon-assigned event sequence).

## 12. Push Notifications (Web Push API + VAPID)

### Why in v0.1

Commute use case requires "agent finished, look at phone" pings.

### Approach

- Daemon generates VAPID keypair on first run, stored in `~/.tether/users/default/vapid.json`
- PWA's service worker subscribes to push using VAPID public key
- Daemon persists subscription per device + (optionally) session
- On trigger event, daemon signs short payload with VAPID private key, POSTs to subscription endpoint
- Browser delivers as system notification

### Trigger events (v0.1)

- CC requests permission for tool call (high priority)
- CC indicates completion (idle for 30s after responding) (medium priority)
- CC errors / crashes (high priority)
- Plan-mode reaches review state (medium priority)

User configures which events trigger via PWA settings.

### Payload (deliberately metadata-only)

```json
{ "v": 1, "type": "tool-permission", "sessionId": "ABC", "timestamp": 1714000000000 }
```

NO message content, NO file paths, NO code.

### Subscription lifecycle

- Subscriptions expire (browser-dependent): tether-cli's push send returns 410 → mark subscription invalid → PWA re-subscribes on next open
- PWA reinstall / browser data clear → subscription invalid → re-pair restores it
- `tether subscriptions` lists active push subscriptions

### Documented limitations (in PWA onboarding screen)

> **Notification reliability:**
> - Android Chrome: typically delivered within 1–5 seconds. Reliable.
> - iOS Safari: requires the PWA to be installed via "Add to Home Screen". Even then, notifications may be delayed by 1–10 minutes during low-power mode, or rarely silently dropped.
> - Tether will indicate connection status. If you are waiting for a critical event, do not solely rely on push — keep the PWA open or check periodically.

### Cost

Adds ~1 week to v0.1 timeline.

## 15. Deployment & CLI Commands

### Daemon deployment (no systemd required)

`tether daemon` is a userspace Node.js process. No root, no systemd, no privileged container access required.

#### Recommended patterns (in priority order)

##### A. Container CMD/ENTRYPOINT (you control container build)

```dockerfile
FROM node:20-alpine
RUN apk add --no-cache python3 build-base   # for node-pty build
RUN npm install -g @xiaokangw/tether
EXPOSE 7777
ENTRYPOINT ["tether", "daemon"]
```

Container restart policy = daemon restart policy. Cleanest for production. **Strongly preferred over `--detach` in container environments.**

##### B. tmux/screen (you only have SSH, can't change container)

```bash
tmux new -d -s tether 'tether daemon 2>&1 | tee -a ~/.tether/daemon.log'
```

##### C. Built-in `--detach` (with healthcheck) — fallback for non-container hosts

```bash
tether daemon --detach
# Internal: double-fork + setsid
# Parent waits for child SIGUSR1 (sent after daemon successfully bound port + restored sessions)
# If child exits or fails before SIGUSR1: parent prints error, returns non-zero
# Pidfile written to ~/.tether/daemon.pid

tether daemon status
tether daemon stop
tether daemon restart
```

**Caveats** documented in --help:
- In containers with PID 1 init issues, prefer ENTRYPOINT (option A)
- Stale pidfile cleanup: `tether daemon stop` removes pidfile; if process died abnormally, pidfile may be stale and `tether daemon status` reports "stale pidfile, remove with --force"

##### D. PM2 / forever (if already in use)

```bash
pm2 start "tether daemon" --name tether
pm2 save
```

### Inside `tether attach`: special key sequences

| Sequence | Effect |
|---|---|

## 16. Authorization

After pairing, what can a client do?

| Capability | v0.1 scope | v0.2+ |
|---|---|---|
| List sessions | All sessions on daemon | Only sessions owned by user |
| Spawn session | Yes, in user's filesystem | Subject to role |
| Attach to session | Any session | Owned only |
| Send input | If holds write lock | Same |
| Approve tool calls | If holds write lock | Same |
| **Read files via RPC** | **Default: only paths within session.cwd subtree.** Override possible by user explicitly setting `--allow-paths <pattern>` on `spawn`. | Restricted by ACL |
| Set mode | If holds write lock | Same |
| Kill session | Any session | Owned only |
| Revoke own device | Yes | Yes |
| Revoke other devices | Yes (single-user) | Owner / admin only |

### File RPC scoping (review #2 must-fix #11)

`session.readFile { sessionId, path }`:
1. Resolve `path` relative to `session.cwd` (the working directory at spawn time)
2. **Canonicalize via `realpath()` — resolve all symlinks to their final target**
3. **Authorization is based on the FINAL canonical path, not on intermediate symlink hops.** A symlink chain `cwd/a → /tmp/b → cwd/c` resolves to a path inside `cwd` and is allowed. A symlink `cwd/dotfile → ~/.bashrc` resolves outside `cwd` and is rejected (unless `~/.bashrc`'s parent is in `--allow-paths`).
4. If canonical path is NOT a descendant of `session.cwd` AND not under any `--allow-paths` root: reject with `rpc-error { code: 'path-out-of-scope' }`
5. **Mid-session mutability:** v0.1 does NOT support modifying `--allow-paths` after spawn. To grant additional access, kill and re-spawn the session. v0.2+ may add `tether session allow-paths add <path>` command.

This prevents the most likely accidental exfiltration: a malformed path or UI bug reading `~/.ssh/id_rsa` or `.env` from outside the project.

### `--allow-paths` semantics

**v0.1 model: comma-separated absolute root prefixes** (NOT glob patterns). A path is permitted if its canonical form starts with any of the configured root prefixes (after canonical equals or descendant check).

```bash
tether spawn ~/projects/foo --allow-paths /home/user/projects/shared,/home/user/.config/foo
# Mobile can read files under ~/projects/foo, ~/projects/shared, ~/.config/foo
# Mobile CANNOT read ~/.ssh, ~/.aws, etc.
```

Glob support is deferred to v0.2+ if there's demand.

### Common UX edge cases (documented for setup guides)

- **Dotfiles symlinked into project (e.g., `cwd/.bashrc → ~/dotfiles/.bashrc`):** rejected by default; user adds `~/dotfiles` to `--allow-paths` if desired.
- **Hoisted node_modules in monorepo (e.g., `cwd/node_modules → ../shared-node-modules`):** if shared dir is outside cwd, rejected; add it via `--allow-paths`.
- **Build output symlinked to `/tmp`:** rejected; add `/tmp/myproject-build` if needed.

## 18. Update Strategy

### Server (tether-cli)

- Distributed via npm: `npm install -g @xiaokangw/tether`
- Update: `npm update -g @xiaokangw/tether && tether daemon restart`
- Migrations in `tether-cli/src/storage/migrations/` run on first start of new version
- Sessions persist across daemon restarts (CC processes get re-parented or respawned via PoC #3 mechanism)

### PWA

- Service worker uses `skipWaiting()` + `clientsClaim()` to make new bundle effective immediately on next page load
- IndexedDB schema versioned; migrations on PWA boot
- Bundle versioned: `/app?v=20260426.1`
- On version skew with server, PWA shows in-app message with "Update available, tap to refresh"; refresh forces service worker activation

### Protocol versioning & compatibility

**Compatibility matrix (review #2 must-fix #13):**

```
tether-cli vX.y compatibility:
  - Accepts PWAs running protocol v[X-3..X]    (= last 3 minor versions back)
  - On v < X-3: server returns auth-error { code: 'protocol-too-old' }; PWA shows
    "Server requires update; refresh PWA" with refresh button
  - On v > X: server returns auth-error { code: 'protocol-too-new' }; PWA shows
    "Server is older than client; ask admin to update server"
```

This prevents the "remote-only user locked out" failure mode: even after a server update, PWAs running protocol versions back 3 minor versions will still connect and show an explicit update prompt. The user can refresh to update.

**Hard breaks (require coordinated rollout):**
- Major version bumps (v: 1 → v: 2) reserved for v0.2 multi-user; we will support v: 1 for at least 1 year for self-hosted deployments

### Daemon-PWA bundle synchronization (default, NOT optional)

**Default for self-host:** daemon's static HTTP server serves the PWA bundle that matches its own tether-cli version (PWA assets are bundled into the npm package). Users who update tether-cli automatically get the matching PWA bundle on next page load. This is the **expected primary path**; it bypasses npm/CDN drift for self-hosters and dramatically narrows the version-skew failure surface.

Users running an externally-hosted PWA (e.g., GitHub Pages) against their own daemon can still encounter skew; the N-3 compatibility window above is for that case.
